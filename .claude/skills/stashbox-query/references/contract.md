# `stashbox` CLI — deep driving contract

Self-contained reference for an agent driving the read-only `stashbox` binary. Where this and any other doc disagree, the binary's own `stashbox catalog` and `--help` win; where a doc and the Go source disagreed, the source won and is reflected here. Verified against source at commit `0721a8a`, binary `v0.1.1`, catalog `schemaVersion: v0.10.0`.

`stashbox` is a stash-box GraphQL client. It is **read-only**: the typed surface is generated from a query-only view of the vendored SDL, so there is no mutation, no subscription, no upload, no async job, and no destructive gate — every command is a query, idempotent, safe to repeat.

---

## 1. Invocation & configuration

Command shape: `stashbox <resource> <verb> [flags]`. Binary resolution: `stashbox` on `PATH` (Homebrew) else `./bin/stashbox` (local build).

Global flags (inherited by every subcommand):

| Flag | Env fallback | Default | Purpose |
|---|---|---|---|
| `--url` | `STASHBOX_URL` | none | Base UI URL of the stash-box instance. |
| `--api-key` | `STASHBOX_API_KEY` | none | Credential, sent in the `ApiKey` header. |
| `-o, --output` | — | `json` | `json` or `table`. |
| `--input` | — | none | Variables: JSON file path, or `-` for stdin. |
| `-h, --help` | — | — | Local help on every node. |

`-v/--version` is **root-only** — never appears in a subcommand's global flags.

**Precedence** for `--url` and `--api-key`: explicit flag → env var → (none). A flag is applied only when non-empty, so an empty flag falls through to the env var.

**`STASHBOX_URL` normalisation** (`normaliseURL`): trims whitespace; requires an `http`/`https` scheme and host; trims a trailing `/`; then appends `/graphql` unless the path already ends in `/graphql`; drops query string and fragment. So `https://stashdb.org` → `https://stashdb.org/graphql`; `https://stashdb.org/graphql` unchanged; `https://host/foo` → `https://host/foo/graphql`. Posting GraphQL to the bare base returns the SPA HTML, which is why the suffix is mandatory. If neither flag nor env supplies a URL, any server command short-circuits to `ErrNoURL` → `usage` (exit 2) **before** any network I/O.

**`STASHBOX_API_KEY`** is injected as the `ApiKey` HTTP header. It is optional — when empty, no header is sent (some instances allow unauthenticated reads). **Prefer the env var over `--api-key`:** a value on the command line is exposed in the process listing (`ps`/`/proc`) and shell history. cobra never echoes a string flag's value in usage/error text, so a malformed invocation does not leak the key.

Truly offline subcommands (never touch the network): `catalog`, `help`, `completion`. **`config get` is online** — it is the `GetConfig` GraphQL query against the connected server.

## 2. The catalog — authoritative, offline, exhaustive

`stashbox catalog [OpName]` prints the build-time embedded operation catalog. No server connection needed (it is `//go:embed`ed `catalog.json`). With no argument it emits the whole document; with an op name, just that entry. An unknown name → `usage` (exit 2): `{"code":"usage","message":"no operation \"X\" in the catalog","retryable":false}`.

Document structure (three top-level keys):
- `schemaVersion` — the vendored SDL version the surface was generated from (`"v0.10.0"`).
- `commands` — **37** entries, one per query root field. Each: `field` (GraphQL field name), `kind` (always `"query"`), `args` (array of `{name, type, required}`, with `default` where the schema sets one; the `type` string preserves `!`/`[]`; omitted when the op takes no args), `returnType` (base named type), `exitCodes` (the per-op subset), and `deprecated` (present only on deprecated ops, carrying the reason string).
- `$defs` — **47** entries: **25 input objects** (`kind:"input"`, each with `fields:[{name,type,required}]`) and **22 enums** (`kind:"enum"`, each with `values:[{value,description}]`).

**The catalog is the exhaustive index; `docs/cli/` is not.** The Markdown under `docs/cli/` is generated by cobra's `doc.GenMarkdownTree`, which skips commands cobra reports unavailable — and a `.Deprecated` command is unavailable to the doc tree. So `docs/cli/` omits the two deprecated ops while they remain invokable as hidden `search-one` leaves (see §3). Always use `catalog` as the complete index.

Use `catalog` before building any `--input` or choosing any enum. The `$defs` dictionary is the only authoritative source of exact field names and enum symbols.

That is **37 reachable leaves — one per catalog op.** 35 are listed in their parent's help; the **2 deprecated ops are reachable as hidden leaves** — `performer search-one` (`SearchPerformer`, returns bare `Performer`; → use `searchPerformers`) and `scene search-one` (`SearchScene`, returns bare `Scene`; → use `searchScenes`). Both are invokable with the same `--term`/`--limit` flags but carry cobra's deprecation banner and are omitted from the help tree and from `docs/cli/`. The non-deprecated plural `search` verbs are the current path. `completion {bash,fish,powershell,zsh}` exist with a `--no-descriptions` flag (shell autocompletion, not data).

Notable asymmetries:
- `site-category get --id` binds `Int!`; every other `get --id` binds `ID!`/`ID`.
- `studio get` / `tag get` / `user get` are "or" finds (both selectors nullable) — supply exactly one.
- search-verb mapping: `scene search`→`SearchScenes`, `performer search`→`SearchPerformers` (paginated, return `Query…ResultType`); `studio search`→`SearchStudio`, `tag search`→`SearchTag` (singular ops, bare return type). All four take `--term`/`--limit`. The hidden `search-one` leaves wrap the singular deprecated `SearchScene`/`SearchPerformer`.

**Zero-arg ops take no variables; a valid `--input` object is silently ignored (NOT an error):** `site query`, `site-category query`, `tag-category query`, `draft list`, `user me`, `misc version`, `config get`, `notification unread-count`. A well-formed JSON-object body's keys are simply never sent (verified: such a body on `misc version`/`draft list` builds and dispatches the request exactly as the no-input baseline does). Only a malformed/non-object/unparseable `--input` errors, as `usage` (exit 2) — and that check applies to every op, not just zero-arg ones.

## 4. The `--input` / convenience-flag model

**Raw-JSON passthrough, never decoded through a typed Go struct.** `--input <file>` or `--input -` (stdin) is `json.Unmarshal`ed into `map[string]json.RawMessage`; values stay `json.RawMessage` and bind onto the wire request unmodified. Consequence, guaranteed by type: a present field stays, an omitted field stays omitted, an explicit `null` stays `null` — byte-for-byte. The top level **must be a JSON object**; a non-object or unparseable body → `usage` (exit 2). Empty/absent `--input` → empty map. There is **no arg-count guard**: extra keys (including any key on a zero-arg op) are forwarded as variables, not rejected.

Top-level keys are the **GraphQL variable names**. For a `*Query*` op that is the single `input` object (so the file is `{"input": { … }}`); for `scene get` it is `id`; for `list-by-fingerprints` it is `fingerprints`.

**Convenience flags** — exactly five, each binding one variable:

| Flag | Binds | Encoding |
|---|---|---|
| `--id` | `id` | JSON string |
| `--name` | `name` | JSON string |
| `--username` | `username` | JSON string |
| `--term` | `term` | JSON string |
| `--limit` | `limit` | bare JSON number; non-integer → `usage` (2) |

Rules (all source-verified):
- A flag is **registered on a leaf only when the op declares that scalar arg** (looked up in the embedded catalog). A flag can never inject an arg the op does not accept.
- Flags merge **under** `--input`: any `--input` key wins; a flag only fills an arg `--input` did not set, and only when the flag was changed.
- The dominant `input` object argument is supplied through `--input` only, never a flag.

## 5. Query / filter / paging / enum model

### Flat filter, implicit AND — no boolean envelope

Every `*QueryInput` is a single flat object. There is **no `AND`/`OR`/`NOT`** combinator and **no nested/recursive depth** (confirmed from `schema/types/filter.graphql` and every `*QueryInput`). Supplied fields are implicitly conjoined (AND); per-field semantics come from each criterion's `modifier`. The only "NOT" tokens in the schema are the `CriterionModifier` symbols `NOT_EQUALS`/`NOT_NULL`, not logical operators. Free-text `String` fields are "like" matches unless the value is quoted.

### Three field shapes

1. **Plain scalar / enum** — e.g. `title`, `names`, `name`, `url`, `parentStudio`, `gender`, `has_parent`, `is_favorite`, `category_id` (single `ID`), `performed_with`/`studio_id` (single `ID`). Passed as a bare JSON value.
2. **Scalar criterion** — `{ "value": …, "modifier": <CriterionModifier> }`:

   | Input type | `value` type |
   |---|---|
   | `IntCriterionInput` | `Int!` |
   | `StringCriterionInput` | `String!` |
   | `DateCriterionInput` | `Date!` |
   | `MultiStringCriterionInput` | `[String!]!` |
   | `EyeColorCriterionInput` | `EyeColorEnum` |
   | `HairColorCriterionInput` | `HairColorEnum` |
   | `BreastTypeCriterionInput` | `BreastTypeEnum` |
   | `RoleCriterionInput` | `[RoleEnum!]!` |
   | `BodyModificationCriterionInput` | compound: `location: String`, `description: String` (no single `value`) |

3. **Relationship criterion** — `MultiIDCriterionInput` (`value: [ID!]`, nullable list) or `IDCriterionInput` (`value: [ID!]!`), plus a `modifier`. Used for `tags`, `performers`, `studios` (`MultiIDCriterionInput`) and `parent` (`IDCriterionInput`).

### CriterionModifier (the full set, verbatim)

`EQUALS` (=), `NOT_EQUALS` (!=), `GREATER_THAN` (>), `LESS_THAN` (<), `IS_NULL`, `NOT_NULL`, `INCLUDES_ALL`, `INCLUDES`, `EXCLUDES`.

### Paging & sort

`page` (default `1`) and `per_page` (default `25`) are `Int!` on every `*QueryInput` (catalog marks them `required:false` because the server defaults them). Sort is two flat fields: `direction` (`SortDirectionEnum`: `ASC`/`DESC`) + `sort` (the per-entity `*SortEnum`). Per-entity defaults: scenes `DATE`/`DESC`, performers `CREATED_AT`/`DESC`, studios `NAME`/`ASC`, tags `NAME`/`ASC`.

Result wrappers carry a non-null total `count` (independent of the returned page) plus the row list — use `count` to drive paging:
- `QueryScenesResultType { count, scenes }`
- `QueryPerformersResultType { count, performers, facets }` (`facets` only populated for search)
- `QueryStudiosResultType { count, studios }`
- `QueryTagsResultType { count, tags }`

### The query inputs (field → classification)

**`SceneQueryInput`:** `text` `String` *(deprecated → use `title`)*; `title` `String`; `url` `String`; `date` / `production_date` `DateCriterionInput`; `studios` / `tags` / `performers` `MultiIDCriterionInput`; `parentStudio` `String`; `alias` `StringCriterionInput`; `fingerprints` `MultiStringCriterionInput`; `favorites` `FavoriteFilter`; `has_fingerprint_submissions` `Boolean`; `page`/`per_page` `Int!`; `direction` `SortDirectionEnum!`; `sort` `SceneSortEnum!`.

**`PerformerQueryInput`:** `names` `String`; `name` `String`; `alias` `String`; `disambiguation` `StringCriterionInput`; `gender` `GenderFilterEnum`; `url` `String`; `birthdate`/`deathdate` `DateCriterionInput`; `birth_year`/`age`/`height`/`band_size`/`waist_size`/`hip_size`/`career_start_year`/`career_end_year` `IntCriterionInput`; `ethnicity` `EthnicityFilterEnum`; `country`/`cup_size` `StringCriterionInput`; `eye_color` `EyeColorCriterionInput`; `hair_color` `HairColorCriterionInput`; `breast_type` `BreastTypeCriterionInput`; `tattoos`/`piercings` `BodyModificationCriterionInput`; `is_favorite` `Boolean`; `performed_with`/`studio_id` `ID`; `page`/`per_page` `Int!`; `direction` `SortDirectionEnum!`; `sort` `PerformerSortEnum!`.

**`StudioQueryInput`:** `name` `String`; `names` `String`; `url` `String`; `parent` `IDCriterionInput`; `has_parent` `Boolean`; `is_favorite` `Boolean`; `page`/`per_page` `Int!`; `direction` `SortDirectionEnum!`; `sort` `StudioSortEnum!`.

**`TagQueryInput`:** `text` `String`; `names` `String`; `name` `String`; `category_id` `ID`; `page`/`per_page` `Int!`; `direction` `SortDirectionEnum!`; `sort` `TagSortEnum!`.

`EditQueryInput`, `ModAuditQueryInput`, `QueryNotificationsInput`, `UserQueryInput`, the `*Existing*` and fingerprint inputs: read their fields from `stashbox catalog <Op>` + `$defs`.

- **EyeColorEnum:** `BLUE`, `BROWN`, `GREY`, `GREEN`, `HAZEL`, `RED`
- **HairColorEnum:** `BLONDE`, `BRUNETTE`, `BLACK`, `RED`, `AUBURN`, `GREY`, `BALD`, `VARIOUS`, `WHITE`, `OTHER`

The metadata values themselves — performer names, scene titles, tag values (sex acts, body parts) — are literal data. Pass and report them verbatim; you can't match or query a title or tag you won't name, so do not euphemise, abbreviate, or soften a value you query or surface.

## 6. Exit-code taxonomy

Frozen `(name → integer)` pairs, defined once in `internal/exitcode`. The envelope's `code` is the name; the process exit status is the integer; the two always agree.

| Code name | Exit | Emitted by this client? |
|---|---:|:---:|
| `ok` | 0 | yes (no envelope) |
| `internal` | 1 | yes — reserved catch-all (never advertised in the catalog) |
| `usage` | 2 | yes |
| `auth` | 3 | yes |
| `transport` | 4 | yes |
| `validation` | 5 | yes |
| `server-fault` | 6 | yes |
| `not-found` | 7 | yes, but only on a not-found-*shaped* server error — a plain missing record is `null`/exit 0 (see note) |
| `destructive-refused` | 8 | **never** (vocabulary parity only) |
| `job-failed` | 9 | **never** |
| `still-running` | 10 | **never** |
| `unconfirmed` | 11 | **never** |

Codes 8–11 are structurally unreachable: this binary has no write/job/destructive path (no `ExitDestructiveRefused`/`ExitJobFailed`/`ExitStillRunning`/`ExitUnconfirmed` vars, no `classifyExit` branch returning them). They stay defined so the vocabulary matches sibling read/write clients.

**A missing record is `null` at exit 0 — not `not-found`.** Every single-entity find (`scene`/`studio`/`tag`/`user`/`site-category`/`edit`/`draft get`, and `user me`) returns a *nullable* type, so the generator advertises `not-found` (7) on each — but a well-formed id that matches nothing comes back as **successful data** `{"find<X>": null}` with **exit 0** (verified live: `scene get --id <valid-but-absent-uuid>` → `{"findScene":null}`, exit 0, empty stderr). `not-found` (7) fires only when the server actually emits a GraphQL error whose message reads like a missing object (the §7 `not found`/`does not exist`/`no such` match) — not for a plain missing-id lookup. To detect absence, test for `null` data, never a non-zero exit.

What classifies as `usage` (exit 2, all caught before any request): unknown subcommand, unknown `--output`, unknown `catalog` op name, bad/non-object `--input`, non-integer `--limit`, and a missing URL (`ErrNoURL`). `1` (`internal`) is the catch-all and is deliberately absent from the catalog's advertised codes.

## 7. Error envelope

On failure the CLI writes **one compact, single-line, newline-terminated JSON object to stderr**, then exits.

| Field | JSON tag | Type | Always present? |
|---|---|---|---|
| `code` | `code` | string | **yes** — the taxonomy **name** (always matches the exit integer's class) |
| `message` | `message` | string | **yes** |
| `graphqlErrors` | `graphqlErrors,omitempty` | string[] | when applicable |
| `field` | `field,omitempty` | string | **never** — nothing in this binary sets it; treat as reserved/absent |
| `retryable` | `retryable` | bool | **yes** (no omitempty — serialized even when `false`) |

`retryable` is `true` only for a transient `*TransportError` (network/timeout, 5xx/429); it is forced `false` whenever the error is `ErrUnauthorized`, and is `false` for GraphQL/auth/client errors. Never blind-retry a `false`. `graphqlErrors` is usually a GraphQL-error thing but can also appear on a `transport`-classified envelope when a non-2xx HTTP response embedded a GraphQL `errors[]` array.

Example (not-found GraphQL error):
```json
{"code":"not-found","message":"stashbox: graphql error: scene not found","graphqlErrors":["scene not found"],"retryable":false}
```

### Classification order (most-specific wins)

Stage 1 turns the raw error into a typed value: an HTTP-200-with-`errors[]` → `*GraphQLError` (wrapped as unauthorized if its `extensions.code` ∈ {UNAUTHENTICATED,UNAUTHORIZED,FORBIDDEN} or its message looks auth-ish); a non-2xx → `*TransportError{StatusCode}` (401/403 also joined with `ErrUnauthorized`); anything else (network/decode/cancelled context) → `*TransportError{StatusCode:0}`.

Stage 2 (`classifyExit`) buckets, in this exact order:
1. `*usageError` → **usage** (2)
2. `ErrNoURL` → **usage** (2)
3. `ErrUnauthorized` → **auth** (3) — **wins over transport and over any GraphQL-message bucket**, so a 401/403 lands on `auth`, not `transport`.
4. `*TransportError` → **transport** (4) — non-2xx, network failure, cancelled context.
5. `*GraphQLError` → message-bucketed:
   - **not-found** (7) if any message (lowercased) contains `not found` / `does not exist` / `no such`;
   - else **validation** (5) if any contains `validation` / `invalid` / `must be` / `is required` / `cannot be null`;
   - else **server-fault** (6).
6. anything else → **internal** (1).

The not-found/validation bucketing is a message heuristic (stash-box returns plain-English resolver errors with no machine `extensions.code`).

## 8. Output formats

`-o/--output`, set is exactly `{json, table}`:

- **`json` (default, the machine format).** Pretty-printed, 2-space indent, trailing newline, **no TTY detection** (stable whether or not stdout is a terminal). Empty/`null` data renders the literal `null`, so output is always a parseable document.
- **`table` (best-effort, human).** `text/tabwriter`: one row per item for a list result (a column per scalar key, sorted union across items), or a two-column KEY/VALUE table for a single object. Nested objects/arrays collapse to the placeholder `{…}`; missing keys render blank. The machine format is `json` — drive automation off `json`.

An unknown `--output` is a `usage` error (exit 2), caught before any request. There is no `ndjson`/`yaml`.

## 9. Secret redaction

Two independent redactors, one per stream.

**STDOUT response redactor** (runs for both `json` and `table`, before any byte is written):
- An `api_key` / `apiKey` object field → whole value replaced with `"REDACTED"`, regardless of shape. (stash-box's `User.api_key` is exposed to its owner, so `user me`/`get`/`query` can carry your own key.)
- URL query creds masked — any string value scrubbed for `apikey`/`api_key`/`token`/`access_token` query params → `REDACTED`.
- A decode/encode failure returns an error rather than printing unredacted.

**STDERR envelope redactor** masks only the same four **query-param** creds in `message` and each `graphqlErrors` entry. It does **not** do a JSON-field walk, so the `api_key`-field replacement is **stdout-only** (moot in practice — the envelope is a flat error string). Note: `user:password@host` userinfo is **not** masked by either of these two redactors (only by a separate URL-config/log path); avoid putting credentials in URL userinfo.

`--api-key` value never appears in cobra usage/error text.

## 10. Idempotency & being a good guest

Every command is a query; none changes server state; all are safe to repeat. StashDB is a shared, community-run public service — keep concurrency bounded, do not hammer it, and do not mask a failure with a blind retry. Use the envelope's `retryable` to decide whether a retry could even help (only transient transport faults), and back off rather than loop. The same binary works against any other stash-box via `--url`.

## 11. Validated recipes

Shapes validated against the catalog (`schemaVersion v0.10.0`); not executed against a server.

```sh
stashbox user me                                   # Me — auth check, zero args
stashbox scene get --id <scene-uuid>               # FindScene
stashbox performer search --term "Riley Reid" --limit 5   # SearchPerformers
stashbox edit get --id <edit-uuid>                 # FindEdit
```

`QueryScenes` — scenes with all of several tags, paged (`scene query --input f.json`):
```json
{ "input": {
  "tags": { "value": ["<tag-uuid-1>", "<tag-uuid-2>"], "modifier": "INCLUDES_ALL" },
  "page": 1, "per_page": 25, "sort": "DATE", "direction": "DESC"
```
